《流畅的python》读书笔记（四）

第4章文本和字节序列，读完这一章数据结构就算弄完了。看目录这一章的内容都不太熟悉，估计需要比较长的时间来读。

第4章 文本和字节序列

python3明确区分了人类可读的文本字符串和原始的字节序列。

字符问题

unicode中把字符的标识和字节表述进行了区分。
字符的标识，既码位，是0~1114111的数字，在unicode中以4-6个16进制的数字表示。
字符的具体表述取决于所用的编码，编码是码位和字节序列之间转换用的算法。
关于“编码”，有两种理解，一种是把码位转化成字节序列，另一种是把字符串转换成计算机存储的字节序列。输出都是字节序列，两者的术语场景不用。

字节概要

python内置了两种不同的二进制序列类型，python3里的不可变的bytes类型和python2.6引进的可变的bytearray类型。
bytes和bytearray对象的各个元素是介于0-255之间的整数，但是其切片还是二进制序列。
所以这一节主要是在讲二进制序列类型。

基本的编解码器

python内置了上百种编解码器。
读到这里我才明白所谓的编码就是把文本（字符串）转换成码表中各编码方式对应的码位。

了解编码问题

把文本转换成字节序列时，如果目标编码中没有的定义某个字符，那就会抛出UnicodeEncoderError异常，除非把errors参数传给编码方法或函数。

把字节序列（二进制序列）转换成文本时，遇到无法转换的字节序列，会抛出UnicodeDecodeError。

python3默认使用UTF-8编码源码，python2默认使用ASCII。
如何找出字节序列用的那种编码？简单地说，不能，必须有人告诉你。
可以用chardet来检测，命令行工具是chardetect。
UTF-16LE是小字节序，UTF-16BE是大字节序。

处理文本文件

要理解unicode三明治模型。
locale.getpreferredencoding()返回的是最重要的，是打开文件的默认编码，也是重定向到stdin/stdout/stderr。
最好别依赖默认编码。
sys.getfilesystemencoding()用于编解码文件名。

为了正确比较而规范化Unicode字符串

因为unicode有组合字符（变音符号和附加到前一个字符上的记号），所以字符串比较起来很复杂。
例如e\u0301和e’(不知道那个符号要怎么打，cafe的最后一个字符)是一样的，但是他们的unicode不一样，所以要用某种方法进行规范化，例如,normalize(‘NFC’,s1)。
规范化的形式包括NFC, NFD， NFKC, NFKD。（不常用，用的时候再查吧。）

Unicode文本排序

讲的是比较非ascii码时会遇到的问题。我暂时还用不到。

Unicode数据库

unicode数据库记录了字符是否可答应、是不是字母、是不是数字等，有isidentifier/isprintable/isdecimal/isnumeric。
分数也是isnumerica,但不是isdigit。

支持字符字符串和字节序列的双模式API

正则表达式中，字符串类型的pattern，前面加r,如r’\d+’，字节序列类型的pattern，前面加rb，如rb’\d+’。
可以使用正则表达式搜索字符串和字节序列，但是在后一种情况总，ascii范围外的字节不会当成数字和组成单词的字母。
os模块中的所有函数、文件名或路径名参数既能使用字符串，也能使用字节序列。
fsencode(filename)和fsdecode(filename)是os模块提供的特殊编解码函数，依赖sys.getfilesystemencoding()。
